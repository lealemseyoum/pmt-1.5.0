Introduction

Why is picture metadata important for imaging?

Picture metadata are used to convey additional information about the picture-the scene, the camera, the people in the picture, or other related data. More powerful imaging systems are created through understanding and using this additional information, rather than simply passing around pixels. For example, picture metadata describing the scene content are used to provide easier and more useful image organization tools for consumers-allowing consumers to capture more of their memories. Picture metadata describing the image creation and image processing history are used to optimize display and printing.

What is the Picture Metadata Toolkit?

For the full benefits of picture metadata to be realized across the open system, imaging applications first need a universal means to define metadata, and a means to create, access, update and save metadata associated with pictures. The Picture Metadata Toolkit addresses this need.

Initially available for C++ developers on the Windows and Unix platforms, this toolkit provides software developers uniform access for to a variety of metadata storage formats, including TIFF (several variants), Exif 2.0 and XML encoding.

Toolkit features

  • Source code available under Open Source License enables users to contribute to the toolkit
  • Open Source License allows for toolkit usage in commercial applications
  • Foundation for the native support of the DIG 35 picture metadata standard
  • Common interface to metadata in different file formats (Exif 2.0, TIFF, and XML encoding currently supported)
  • Enables copying standard metadata from input source to output destination
  • Full-featured metadata definition language, with local metadata definition capability
  • Easy exchange of metadata between heterogeneous systems via XML encoding
  • Uses OpenTiff, an Open Source object oriented toolkit for accessing TIFF image files
  • Includes a toolkit for accessing Exif images (binary distribution only). In the very near future, Exif support will be provided through OpenExif, an Open Source object oriented toolkit for accessing Exif Image files.

Distribution Directory Structure

In the description below, $PMTROOT is the placement of the "Pmt" directory in your file system, e.g., C:\Pmt.

The following directories and files reside in $PMTROOT:
DefaultDefinition Directory holding the file containing the default metadata definitions
config-scripts/ Directory of files containing M4 Macros needed by the GNU auto configuration facility
doc/ Directory holding the documentation
doc/html/ Directory holding the API documentation in HTML format
lib Directory contains the libraries for static/dynamic linking 
projects/ Directory holding the build files for platforms other than Unix / Linux
scripts/ Directory holding script files for automatic documentation generation and building releases
src/ Directory holding the source code files
test/ Directory holding the standard tests
Makefile.in, Makefile.am Makefiles used by the auto configuration facility to generate the final Makefile. WARNING: Makefile.in's are automatically generated by 'automake' from the corresponding Makefile.am's. Do not edit Makefile.in's.
README.config A readme file for PMT's developers providing some background on how the auto configuration facility is setup for PMT.
ReleaseNote.txt A compilation of release notes for the current and all previous releases..
aclocal.m4 Contains the M4 macros that are run by the 'configure' script.
bootstrap Convience shell script for PMT's developers. Run this after a change to one of the Makefile.am's.
configure The main configuration script. WARNING: configure is automatically generated by 'autoconf' from configure.in, do not edit.
libtool Used by the Makefiles to build libraries. WARNING: libtool is automatically generated by 'configure', do not edit.
license.html File containing the Kodak Public License
readme.html this file

The $PMTROOT/src directory contains the source code and has the
following subdirectory structure (source code distribution only):
Ek  Generic utility classes. Compiler and platform dependencies control. 
PmtCore  The "core" classes of the toolkit. The PmtMetadata class hierarchy is defined here.
PmtInterpreter  Logical Definition and Logical Instance interpreters and support classes. 
PmtAccessor  PmtAccessor and its implementation classes: PmtTiffAccessor, PmtExifAccessor and PmtXmlAccessor. 
PmtTranslator  PmtTranslator and its implementation classes. 
PmtUtils  Utility classes specific to PMT.

Installation Notes

Win32 Platform

If you are building on the Win32 platform, it is important that you understand a few build details:
(In the description below, $PMTROOT is the placement of the "Pmt" directory in your file system, e.g., C:\Pmt).

DLL directories and the PATH setting: These DLLs are placed in the "$PMTROOT\lib" directory.  You must either add this directory to the PATH environment variable in your system, or copy the DLL files into a directory already in your PATH, but we do not recommend this -- you can wind up with multiple copies of the DLL, and wind up using the wrong one.

To change your PATH:  Do this under NT by using the System control panel. Pick the "Environment" tab, select the variable "Path" in the "System Variables" list.  In the "Value" control, append the $PMTROOT\lib path". Click the Set button, then the Ok button.

Link with Runtime libraries: All the DLLs link with the C runtime library "Debug Multithreaded DLL" or "Multithreaded DLL." (This is changed through the Project Settings dialog, on the C/C++ tab, under Code Generation.) It is important that any executable or other DLL you build which uses the PMT DLLs links with these runtime libraries as well. If you do not do this, you will likely get memory errors when you run the executable.

Build / Install on Win32

In the description below, $PMTROOT is the placement of the "Pmt" directory in your file system, e.g., C:\Pmt.

For full instructions on building PMT, please refer to the PmtUserGuide.doc file in the $PMTROOT/doc directory.

The steps are:
  • Unzip the pmt-src-XXX.zip file
  • Setup the following environment variables to their corresponsing paths as shown below:
    PMT_XERCES_INC_PATH This points to the Xerces include directory. Example setting: C:\Xerces\xerces-c\include
    PMT_XERCES_LIB_PATH This points to the Xerces library directory. Example setting: C:\Xerces\xerces-c\win32\lib
    KIESEXIFROOT This points to the Kies toolkit directory. Example setting: C:\Pmt\Toolkits\Kies
    IJLROOT This points to the IJL toolkit directory. Example setting: C:\Pmt\Toolkits\ijl
    JPEGROOT This points to the JPEG toolkit directory. Example setting: C:\Pmt\Toolkits\Jpeg
    TIFFROOT This points to the TIFF toolkit directory. Example setting: C:\OpenTiff\opentiff-1_0_1-src
  • Start Microsoft Visual C++ 6.0.
  • Choose "File" menu and select "Open WorkSpace".
  • In the file chooser, choose either $PMTROOT\projects\Win32\Vc6.0\BuildAllStatic.dsw or $PMTROOT\projects\Win32\Vc6.0\BuildAllDynamic.dsw. Open this workspace.
  • These workspaces include all the PMT component libraries. The BuildAllStatic.dsw workspace will build all the static libraries associated with PMT. The BuildAllDynamic.dsw workspace will build all the dynamic libraries associated with PMT.
  • Set the active Project. Choose "Project" menu and select "Set active project". In the submenu, select either the "_BuildAllStatic" or "_BuildAllDynamic" project.
  • Set the active configuration ("Win32 Debug" or "Win32 Release") and make sure this matches your PATH setting as described in the previous chapter. (See note below.)
  • Choose "Build" menu and select "Rebuild All". If you want to build the Debug and Release configurations at the same time, choose "Build" menu and select "Batch Build..." instead (and mark all configurations as checked), then click the button named "Rebuild All".
Note: To set the active configuration, two different possibilities are:
  • Choose "Build" menu, select "Set Active Configuration", and select "Win32 Release" or "Win32 Debug".
  • Another way is to select "Customize" in the "Tools" menu, select the "Toolbars" tab, enable "Build" instead of "Build Minibar", and click on "Close". This will bring up a toolbar which you can move aside the other permanent toolbars at the top of the MSVC window. The advantage is that you now have an easy-to-reach pop-up menu that will always show the currently selected active configuration.  Or, you can drag the project and configuration selections and drop them on the menu bar for later selection.

Build / Install on Unix / Linux

For full instructions on building PMT, please refer to the PmtUserGuide.doc file in the $PMTROOT/doc directory.

The Unix / Linux build enviornment uses the GNU auto configuration facility. The `configure' shell script attempts to guess correct values for various system-dependent variables used during compilation. It uses those values to create a `Makefile' in each directory of the package that contains a Makefile.in (and Makefile.am). It also creates a shell script `config.status' that you can run in the future to recreate the current configuration, a file `config.cache' that saves the results of its tests to speed up reconfiguring, and a file `config.log' containing compiler output (useful mainly for debugging `configure').

Note:
  • The default is to build both the static and shared libraries. The building of the shared libraries can be disabled by supplying 'configure' with the '--disable-shared' option. Similarly, the building of static libraries can be disabled by supplying the '--disable-static' option.
  • The default installation root directory is '/usr/local'. The libraries are placed in $root/lib and header files in $root/include. This can be changes by passing 'configure' the '--prefix=/path/to/new/root' option with the right hand side of the assignment being the full path to the new root directory, e.g., /usr/local/pmt/pmt-1.0.2beta.
  • You can remove the program binaries and object files from the source code directory by typing `make clean'. To also remove the files that `configure' created (so you can compile the package for a different kind of computer), type `make distclean'.
